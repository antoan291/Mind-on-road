# Architecture Decision Records

## Purpose

This folder stores ADRs for architectural decisions that should survive across future project work and onboarding.

## Template

- [adr_template.md](adr_template.md)

## When To Create An ADR

Create an ADR when a decision changes or locks in one of these:

- backend architecture or module boundaries;
- database tenancy, constraints, or migration strategy;
- deployment/infrastructure approach;
- security model;
- AI/data handling model;
- major frontend architecture or state-management decisions.

## Rules

- Keep ADRs short, factual, and specific.
- Compare real alternatives and trade-offs.
- Record consequences and follow-up actions.
- Do not invent historical context; mark assumptions explicitly if needed.
